h1. ISO8583 AMEX Decoder – Technical Overview

h2. 1. Purpose

The AMEX ISO8583 Decoder provides a *schema-driven, extensible, and test-friendly* mechanism for parsing AMEX messages into structured Java objects.

It is designed to support:
* Multiple MTI flows (1100, 1110, 1420, etc.)
* Dynamic schema variations
* Composite and nested field structures
* High-quality debugging and observability

---

h2. 2. Architecture Overview

h3. High-Level Flow

{code}
Raw byte[] message
↓
HeaderDecoder (MTI + Bitmaps)
↓
MessageDefinition (resolved by MTI)
↓
FieldDefinition (+ Adjusters applied)
↓
FieldDecoder (Primitive / Composite)
↓
DecodedMessage (internal model)
↓
MessageMapper
↓
Typed Message (e.g. Amex1100Message)
↓
DecodedAmexResult
{code}

---

h3. Core Components

Component  Responsibility
| AmexDecoder | Orchestrates decoding pipeline |
| HeaderDecoder | Parses MTI and bitmaps |
| MessageDefinition | Defines schema per MTI |
| FieldDefinition | Describes field structure |
| FieldDefinitionAdjuster | Dynamically modifies schema |
| FieldDecoder | Parses field values |
| MessageMapper | Maps decoded values to typed object |
| DecodedAmexResult | Final result container |
| DecodedPrettyPrinter | Debug visualization |

---

h2. 3. Core Concepts

h3. 3.1 Schema-Driven Decoding

Each MTI is associated with a *MessageDefinition* that defines:
* Field presence
* Encoding
* Length rules
* Subfield structure

Example:
{code:java}
MessageDefinition<Amex1100Message> definition = new Amex1100Definition();
{code}

---

h3. 3.2 Composite Fields

Complex fields (e.g. DE22, DE55) are defined as:

* CompositeFieldDefinition
* Containing multiple SubfieldDefinition objects

Supports:
* Fixed structure
* Variable length subfields
* Nested decoding

---

h3. 3.3 Dynamic Schema Adjustments

Schema can be modified at runtime using:

*FieldDefinitionAdjuster*

Use cases:
* EMV variants
* Legacy format support
* Conditional fields
* Field renaming or removal

---

h2. 4. DecodeOptions (Dynamic Behavior)

DecodeOptions provide runtime configuration.

h3. Example

{code:java}
DecodeOptions options = DecodeOptions.empty()
.with(DecodeOptionKeys.DE55_VARIANT, "SIMPLIFIED");
{code}

h3. Behavior

Scenario  Result
| No options | Base schema used |
| Options provided | Adjuster modifies schema |
| Adjuster returns null | Base schema fallback |

---

h2. 5. Decoding Usage

h3. Standard Decode

{code:java}
AmexDecoder decoder = new AmexDecoder();

DecodedAmexResult<? extends AmexMessage> result =
decoder.decode(rawMessage);
{code}

---

h3. Decode with Options

{code:java}
DecodeOptions options = DecodeOptions.empty()
.with(DecodeOptionKeys.DE55_VARIANT, "LEGACY");

DecodedAmexResult<? extends AmexMessage> result =
decoder.decode(rawMessage, options);
{code}

---

h3. Typed Decode (Recommended when MTI known)

{code:java}
DecodedAmexResult<Amex1100Message> result =
decoder.decode(rawMessage, new Amex1100Definition(), DecodeOptions.empty());

Amex1100Message message = result.getMessage();
{code}

---

h2. 6. Result Model

h3. DecodedAmexResult

Contains:

Field  Description
| message | Typed POJO |
| flatValues | Debug-friendly key/value map |
| fieldMeta | Offsets, raw hex |
| effectiveFieldDefinitions | Final schema used during decode |

h3. Key Advantage

The result contains a *snapshot of the effective schema*, meaning:
* No need to pass DecodeOptions to downstream components
* Debug tools always reflect actual decode behavior

---

h2. 7. Pretty Printer

h3. Usage

{code:java}
DecodedPrettyPrinter.print(

System.out,

---

h3. Features

* Field highlighting
* Bitmap visualization
* Nested subfield display
* Raw + meta information

---

h3. Highlighting Options

Supports:
* PID format → P2, P55.1
* Field name → PAN, POS_DATA_CODE.CARD_PRESENT

---

h3. Disable ANSI Colors

{code:java}
DecodedPrettyPrinter.setAnsiMode(DecodedPrettyPrinter.AnsiMode.OFF);

{code}

---

h2. 8. Bitmap Handling

Supports:
* Standard binary bitmap
* Escaped byte format (legacy):

{code}
{02} → 0x02
{code}

Handled transparently inside HeaderDecoder.

---

h2. 9. Data Access Strategy

h3. Recommended Usage

Layer  Access Method
| Business Logic | Typed message (Amex1100Message) |
| Debugging | flatValues |
| Schema validation | fieldMeta |

---

h2. 10. Extensibility Guidelines

h3. Adding New Fields

* Update MessageDefinition
* Add SubfieldDefinition if needed

---

h3. Adding Dynamic Behavior

* Create new FieldDefinitionAdjuster
* Attach to field in definition

---

h3. Best Practices

* Keep adjusters small and field-specific
* Avoid modifying decoder core logic
* Prefer schema changes over parsing hacks

---

h2. 11. Example Runner

{code:java}
public class DecodeRunner {

    public static void main(String[] args) {

        byte[] rawMessage = ...;

        AmexDecoder decoder = new AmexDecoder();

        DecodeOptions options = DecodeOptions.empty()
                .with(DecodeOptionKeys.DE55_VARIANT, "SIMPLIFIED");

        DecodedAmexResult<? extends AmexMessage> result =
                decoder.decode(rawMessage, options);

        DecodedPrettyPrinter.print(
                result,
                rawMessage,
                System.out,
                Arrays.asList("PAN", "P22.6")
        );
    }
}
{code}

h2. 12. Summary

The decoder provides:

* Strong separation of concerns
* Runtime schema flexibility
* Safe defaults (no flags = no behavior change)
* Full observability for debugging

It is suitable for:
* Integration testing
* Message validation
* Transaction analysis
* Protocol reverse-engineering
